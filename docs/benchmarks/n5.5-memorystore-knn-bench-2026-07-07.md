# N5.5 — Bench KNN via le chemin `MemoryStore` complet (2026-07-07)

**Contexte** : `docs/TODO-NATIVE-ENGINE.md` N5.5 (barre hardening M6) demande
un bench KNN « via le chemin `MemoryStore` complet, pas l'index nu ». Les
benchs précédents mesuraient une brique à la fois : N3 (`docs/benchmarks/
n3-vector-parity-2026-07-05.md`) mesure `PersistentVectorIndex` seul (insert/
search sur des vecteurs bruts), M6 (`docs/benchmarks/m6-knn-results-2026-07-01.md`)
mesure `vector_top_k` libSQL seul. Aucun des deux ne traverse
`NativeMemoryStore::put_memory`/`recall_vector` — le chemin qu'un vrai
`remember`/`recall` emprunte réellement : oversampling ×8 (ADR-012),
résolution `vec_id → (agent, id)`, hydratation du contenu, filtre agent/
validité/couche, `last_access`. Ce document couvre cette lacune.

**Méthode** : `crates/basemyai/tests/native_memory_store_bench.rs` — même
discipline que `vector_recall.rs`/`vector_bench.rs` (chiffres réels,
chronométrés, jamais supposés ; pas de couche statistique Criterion, un
scénario réaliste imprimé tel quel). `N` mémoires insérées une par une via
`MemoryStore::put_memory` (le vrai chemin `remember`, pas un bulk-load),
384d (parité produit/M6), puis 50 requêtes `recall_vector(k=10)` sur des
vecteurs disjoints des ids insérés.

## Résultat — N=10 000, `--release`

```text
cargo test --release -p basemyai --features test-util,engine-native \
    --test native_memory_store_bench -- --ignored --nocapture
```

```text
native_memory_store_bench: n=10000 dim=384 queries=50 k=10 ->
put_memory total 129.644713s (~12.964 ms/put, full MemoryStore path) ;
recall_vector total 851.5743ms (~17.031 ms/query, oversample×8 + hydrate + touch)
```

## Comparaison honnête

- **Insert** : ~12.96 ms/put via le chemin `MemoryStore` complet (record +
  vecteur + FTS, un seul enregistrement WAL) contre 78–79 ms/ligne pour le
  build incrémental libSQL à 10k (`docs/benchmarks/m6-knn-results-2026-07-01.md`,
  chiffre lui-même un taux de bulk-load, pas un `remember` unitaire réel).
  Cohérent avec, et à l'intérieur de, la fourchette déjà mesurée sur l'index
  nu par N3 (5.7–17.3 ms/ligne, `docs/benchmarks/n3-vector-parity-2026-07-05.md`)
  — le `MemoryStore` complet n'ajoute pas de surcoût significatif au-delà de
  l'index seul.
- **Query** : ~17.03 ms/`recall_vector` (oversampling ×8 + résolution +
  hydratation + filtre + `touch` inclus) contre ~48.98 ms (moyenne) pour le
  `vector_top_k` **nu** libSQL au même N. Le chemin natif fait strictement
  plus de travail par requête et reste ~2.9× plus rapide en bout de chaîne —
  cohérent avec les seuils de requête déjà tenus par l'index nu (7.5/12.7 ms
  à 10k/100k contre un plafond libSQL de ~48-49 ms, N3).

**Limites assumées** : une seule exécution locale, pas de moyenne
statistique (délibéré — voir méthode) ; N=10 000 seulement (le harnais N3
allait jusqu'à 100k sur l'index nu, pas encore refait ici via le
`MemoryStore` complet — item de suivi si un chiffre à 100k devient
nécessaire) ; construit `--release` pour être comparable à M6/N3, alors que
la variante rapide par défaut du gate CI (N=2 000,
`recall_vector_bench_n2000`, `cargo xtask test`) tourne en `dev` non optimisé
et n'est **pas** comparable à ces chiffres — elle sert de signal de
régression continu, pas de mesure de référence.

## Où c'est câblé

- `crates/basemyai/tests/native_memory_store_bench.rs` :
  `recall_vector_bench_n2000` (rapide, `cargo xtask test`),
  `recall_vector_bench_n10000` (`#[ignore]`, manuel — ce document).
- `xtask/src/main.rs` : `TEST` inclut
  `-p basemyai --features test-util,engine-native --test native_memory_store_bench`.
