# N11 — Campagne de durcissement 1M records (2026-07-15)

**Rôle** : le critère de sortie manquant de `PLAN-NATIVE-ENGINE.md` §8 ("campagne
longue régulière... 1M records, compaction répétée... RSS/handles") et le
suivi laissé ouvert par N10 ("benchmark 1M archivé") — jamais réellement
exécuté avant cette session (seul un run partiel, non archivé, avait été
tenté le 2026-07-13, cf. logs retrouvés sur cette machine). Ce document
archive une campagne réelle, mesurée, avec preuve d'intégrité.

- **Outil** : `engine_bench` (binaire `basemyai-engine`), release, workloads
  canoniques identiques à N7.5/N8, **plus un nouveau flag `--verify`** ajouté
  dans cette session (`crates/basemyai-engine/src/bin/engine_bench.rs`) : après
  la fermeture propre de chaque groupe (`kv`, `memory`) et avant la
  suppression du répertoire temporaire, il rouvre le store et appelle
  `basemyai_engine::verify_store(dir, key, VerifyMode::FullLogical)` — le
  mode le plus profond de l'audit ADR-040 §2 (le plan appelle ce mode
  `--deep` ; la CLI/l'API réelles le nomment `FullLogical`/`--logical`, pas
  `--deep` — la campagne utilise le vrai nom). Échec fatal (`fatal(...)`,
  process exit 1) si le rapport n'est pas `healthy`, pour qu'une régression
  silencieuse ne se cache pas derrière un run "vert" par ailleurs.
- **Machine** : Windows x86_64, Intel64 Family 6 Model 186 Stepping 2
  (même laptop que N7.5/N8, comparabilité intra-repo partielle — voir
  "Limites honnêtes" plus bas concernant le bruit de charge machine).
- **Commit** : `0621945`.
- **Campagne réellement exécutée** : **4 cycles** à `n=1 000 000` (dépasse le
  minimum de 3-5 cycles demandé), **clair ET chiffré** à chaque cycle (8
  invocations `engine_bench` au total), plus `cargo xtask
  test-crash-consistency` (kill-loop réel, 7 variantes × 20 cycles). Le
  premier cycle clair (`cycle-1.json`) a tourné avec le binaire **avant**
  l'ajout de `--verify` (build pré-existante au lancement de cette session) —
  c'est la seule des 8 invocations sans audit `verify_store` explicite ;
  toutes les 7 suivantes (`cycle-1.encrypted` à `cycle-4.encrypted`) l'ont.
- **Données brutes** : `docs/benchmarks/data/n11-soak-1m/` — 8 rapports JSON
  (`cycle-{1..4}.json`/`cycle-{1..4}.encrypted.json`, schéma
  `basemyai-engine-bench/1`) + `campaign.log` (sortie brute complète de la
  campagne, y compris les 14 lignes `verify(...)` et les 7 tests
  `test-crash-consistency`).

## RSS : stable à travers les 4 cycles, pas de dérive

| Invocation | RSS peak (Mo) | RSS mean (Mo) |
|---|---|---|
| cycle-1 clear (pré-`--verify`) | 554,5 | 116,4 |
| cycle-1 encrypted | 558,3 | 105,1 |
| cycle-2 clear | 554,4 | 118,5 |
| cycle-2 encrypted | 553,5 | 95,0 |
| cycle-3 clear | 559,2 | 114,5 |
| cycle-3 encrypted | 561,7 | 102,4 |
| cycle-4 clear | 560,6 | 111,8 |
| cycle-4 encrypted | 559,3 | 105,1 |

RSS peak varie entre **553,5 et 561,7 Mo sur les 8 runs — un étalement de
8,2 Mo (1,5 %)**, sans tendance monotone (clair : 554,5 → 554,4 → 559,2 →
560,6 ; chiffré : 558,3 → 553,5 → 561,7 → 559,3 — pas de séquence
strictement croissante dans les deux cas). Cohérent avec la baseline N8
(570 Mo clair / 547 Mo chiffré, même échelle 1M, même laptop) — **aucun
signe de fuite mémoire cycle-à-cycle**. Chaque invocation est un process
`engine_bench.exe` frais (le harnais ne réutilise pas de process entre
cycles), donc ceci mesure "le moteur ne dégrade pas son empreinte mémoire au
fil de runs répétés sur la même machine", pas la RSS interne d'un unique
process de longue durée — la question posée par le plan ("aucune croissance
mémoire inexpliquée") porte spécifiquement sur ce point à cette échelle et
il est vérifié.

## Structure sur disque : identique bit-pour-bit à chaque cycle

Les compteurs `EngineStats` du groupe `kv` (fin de `delete-churn`, juste
avant `flush-compaction`) sont **strictement identiques sur les 4 cycles
clairs entre eux, et sur les 4 cycles chiffrés entre eux** :

| Métrique | Clair (×4 cycles) | Chiffré (×4 cycles) |
|---|---|---|
| `tombstone_count` | 0 | 0 |
| `sst_bytes` | 126 811 236 | 127 190 736 |
| `wal_bytes` | 0 | 0 |
| `compaction_count` (à `kv-fill`) | 249 | 249 |
| `block_cache_hits`/`misses` (à `delete-churn`) | 6 913 / 19 189 | 6 913 / 19 189 |
| `open-large-store` `bytes_read` | 1 708 472 | 1 709 072 |

**`tombstone_count == 0` après `delete-churn`** : les 20 % de clés
supprimées à chaque passe de churn (jusqu'à 3 passes, `compaction_sst_threshold=4`)
sont purgées par une compaction qui suit avant la fin du groupe — comportement
attendu, pas un signe d'accumulation. `wal_bytes == 0` en fin de groupe `kv`
parce que `flush_count` (1000+) a déjà tout vidé vers les SST avant ce point
de mesure ; le groupe `memory` (moins de flushes, écritures composées plus
lourdes) montre un `wal_bytes` non nul (~606 Ko) et **identique aux 4
décimales près sur les 4 cycles clairs** (606 247) et **les 4 cycles
chiffrés** (606 747) — reproductibilité parfaite du workload déterministe,
donc toute divergence future entre cycles serait un signal fort, pas du
bruit.

Reproductibilité parfaite ⇒ le test de non-régression le plus dur qu'on
puisse appliquer ici ("le même workload produit exactement le même état
moteur, cycle après cycle, avec ou sans process frais") est passé
silencieusement sur toute la campagne.

## Chiffré vs clair : surcoût cohérent avec N7.5/N8

`sst_bytes` chiffré = 127,19 Mo vs 126,81 Mo clair (**+0,3 %**, l'enveloppe
crypto par bloc, ADR-030/N8.8) — dans la même fourchette que N8 (127,44 Mo
vs 127,06 Mo à 1M, +0,3 %). Pas de régression du surcoût crypto depuis N8.

## Limites honnêtes (bruit machine, pas caché)

Contrairement à N7.5/N8 (un run par config sur machine calme), **cette
campagne a tourné en tâche de fond pendant que d'autres commandes cargo
(`clippy`, builds) et mes propres boucles d'attente actives (busy-wait pour
observer la progression sans bloquer l'agent) partageaient la même
machine.** Les compteurs structurels (`sst_bytes`, `wal_bytes`,
`tombstone_count`, `compaction_count`, `block_cache_*`, `verify_store`) n'en
sont pas affectés — déterministes, indépendants du CPU disponible — mais les
**latences** le sont clairement :

- `open-large-store` (réouverture froide) : 6,0 à 55,9 ms selon le cycle,
  contre 5,5 ms mesuré en N8 sur machine calme. Pas de tendance
  cycle-à-cycle (6,0 ms au cycle 3 clair, 55,9 ms au cycle 2 chiffré,
  15,4 ms au cycle 4 chiffré) — un pic isolé, pas une dérive. Attribué à la
  contention CPU/disque du moment (confirmé : le cycle 2 chiffré a
  coïncidé avec un `cargo clippy` lancé par l'agent en parallèle), pas à
  une régression moteur.
- `kv-fill` mean : 0,38-0,72 ms/op selon le cycle (N8 : 0,376 ms clair /
  0,388 ms chiffré). Le premier run (`cycle-1.json`, avant même l'ajout de
  `--verify`) est le plus lent (0,72 ms) — cohérent avec un cache disque
  froid en tout début de campagne ; les cycles 2-4 convergent vers
  0,38-0,49 ms, dans la fourchette N8.
- `flush-compaction` p95 : 1,2-2,5 s selon le cycle — pas de baseline N8
  directe à 1M pour ce p95 spécifique (N8 ne le tabule qu'à 100k), donc pas
  de comparaison chiffrée possible ; mentionné pour complétude, pas comme
  régression.

**Aucune de ces variations de latence n'affecte la conclusion de fond**
(pas de fuite, pas de croissance non bornée, verify sain à chaque fois) —
elles auraient nécessité une machine dédiée et calme pour être des mesures
de performance publiables au même niveau de confiance que N7.5/N8. Ce
chantier (N11) porte sur la robustesse/durabilité, pas sur des chiffres de
perf actionnables ; les latences ci-dessus sont donc rapportées pour
transparence, pas comme un nouveau baseline de perf à comparer à N8.

## Ce qui n'a pas été testé (déjà documenté comme hors périmètre)

Comme `soak-campaign.yml` et `docs/status.md` §8.3 le documentaient déjà
avant cette session, et qui reste vrai après : **rotation de clé pendant un
soak, disque presque plein, comptage de handles fichier** — aucune
instrumentation moteur/xtask existante pour ces trois axes. Non fait ici
non plus, pas de fausse promesse. Un comptage de handles fichier serait
approximable indirectement sur Windows via `Get-Process -Id <pid> |
Select-Object HandleCount` en cours de run — non instrumenté dans cette
session (aurait nécessité de modifier le harnais pour échantillonner
pendant l'exécution, pas seulement en fin de workload comme le RSS
sampler actuel) ; candidat de suivi si un futur soak veut fermer ce point
précis.

## Reproduire

```bash
cargo build --release -p basemyai-engine --features test-util --bin engine_bench
for cycle in 1 2 3 4; do
  target/release/engine_bench --n 1000000 --verify --out docs/benchmarks/data/n11-soak-1m/cycle-$cycle.clear.json
  target/release/engine_bench --n 1000000 --verify --encrypted --out docs/benchmarks/data/n11-soak-1m/cycle-$cycle.encrypted.json
done
cargo xtask test-crash-consistency
```

(`--verify` est un flag ajouté dans cette session — voir
`crates/basemyai-engine/src/bin/engine_bench.rs::verify_dir_if_requested`.
`cargo xtask engine-soak <cycles> <n>` fait la même boucle clair+chiffré
automatiquement mais sans `--verify` ni sortie par-cycle nommée — utilisé ici
en partie manuellement pour capturer un JSON distinct par cycle, comme
documenté par la tâche.)
